## TypeScript의 config 속성들
### stict
- TypeScript를 많은 자바스크립트 개발자들이 사용하게 만들기 위해서는 소프트랜딩이 가능하도록 하는 전략이 필요 했을것 같다.
- 만약 strict 를 false로 지정한 경우에는, TypeScript 와 JavaScript 코드간의 공존이 가능하다.
- 즉, 100개를 바꿔야 한다면, 20~30개부터 바꾸면서 시작할 수 있도록 해주는 속성이다.
- 만약 true라면, 엄격하게 TypeScript의 조건을 따르게 된다.
#### noImplicitAny
- strict의 하위에 있는 속성이다.
- 명확하게 타입이 정해지지 않은 타입들은, Any 타입이라고 부르게 된다.
- 만약 noImplicitAny 속성을 true로 지정한 경우에는 Any 타입의 사용을 불허, 즉 명확한 타입을 지정해야만 한다.

### target 과 module
- 컴파일러를 통해 컴파일 되면서 JavaScript 파일을 만들게 되는데, 이때 어떤 버전을 타겟으로 만들지 설정하는 옵션들이다.
- module의 경우 module의 버전을 입력하게 된다.

### sourceMap
- dist라는 폴더에는 TypeScript Compiler가 TypeScript 파일을 Compile한 결과를 출력해놓고 있다.
- 이때 app(어쩌구저쩌구).js 파일과 app(어쩌구저쩌구).js.map 이라는 파일이 생긴다.
- 이 파일을 만들것이다, 생성해라 라는 것이 sourceMap 이다.
- 즉, 개발환경과 실행환경을 일치시킬 수 있게 도와주는 파일이다.

## Type Alias
- 원하는 Type을 만들 수 있는 기능이다. 
- 객체 내부의 Key에 맞는 Value Type을 지정하는 것이 가능해, 원하지 않는 타입이 들어오지 않도록 할 수 있는 장점이 있다.

### Intersection
- 만약 타입끼리 중복되는 부분이 있다면 사용할 수 있는 방식이다.
- 중복된 타입끼리의 Type을 만들고, 이를 & 를 사용해 합치는 방식으로 중복을 없애는 것이 가능하다.
## Type Guard
- 만약 null이 들어갈 수 있는 Type에 어떠한 방어 코드 없이 함수를 실행시킨다면 문제가 발생할 것이다.
- 이때 만약 null인 경우에는 **함수를 진행시키지 말아라**라는 방어 코드가 필요하다.
- 이러한 조건들을 통해 코드가 오작동하는것을 막는 코드를 방어 코드라고 한다.
## Type Inference
- VSCode에서 만약 Type에 대한 추론이 가능하다면 에러 메세지를 발생시키지 않는다.
- 즉, 추론 가능한 Type이 있다면 내부적으로 Type에 대한 설정을 진행한다.

## Generic
- 함수 이름과 함수 인자 사이에 <>를 사용해서 작성한다.
- 어떤 함수에 대한 출력 값의 타입이 여러가지가 있는 경우에 사용하는 함수 입장에서는 어떤 함수가 출력될지 알 수 없다.
- 이를 방어하기 위해서는 대량의 if else 구문을 사용해야 하는데, 이는 굉장히 힘든 작업이 될 수 있다.
- 이때 Generic을 사용하면, 함수를 사용하는 쪽에서 함수를 호출하며 출력되어야 할 타입을 지정해서 사용할 수 있다.

## Interface
- Type Alias와 마찬가지로 Type을 만드는 방법이다.
- type 대신 interface 를 사용해서 작성하며, = 없이 사용한다.
- 가장 큰 차이점은, 타입을 변환시키거나 조합시킬 때이다.
- & 를 사용하는 것이 아닌 extends 를 사용해서 사용한다. (확장된다는 느낌을 강하게 느낄수 있다!)
- 유니온 타입을 사용하지 못한다.

## readonly
- 값을 바꾸지 못하게 만들기 위해서 사용한다.
- 변경이 불가능하도록 만들고 싶은 key 앞에 readonly를 입력해 사용할 수 있다.
- 변경되야 하는 코드와 변경되지 말아야 하는 코드를 미리 정해놓을 수 있기 때문에 코딩 시간을 절약하는것도 가능할것 같다.

## protected
- 하위 클래스가 상속받아서 사용할 목적으로 만든 함수를 노출시키지 않도록 하는 키워드이다.
- 하위 클래스에서만 사용되어야 하기 때문에, 이를 노출시키지 않는것이 바람직하다.

## Mixin
- extends를 사용하게 되면 상속 관계 변경시에 직접 코드를 수정해줘야 하는 문제가 생긴다.
- 즉, A와 B를 상속하고, A와 C를 엮어서 사용하고 싶을때 사용이 불가능하다. (다중 상속을 지원하지 않는다.)
- 이를 위해 Mixin 기법을 사용한다.
- 어려운 기법이기도 하고, 없어도 제대로 동작하는 경우가 많다보니 유연성이 많이 필요한 경우가 아니면 주로 사용되지 않는다. (프로젝트의 성격에 따라서 사용하면 좋을것 같다.)

## abstract method
- 자식에게 메서드를 만들어야한다고 강제할때 사용되는 메서드이다.
- 이를 추상메서드라고 부르며 추상 메서드가 있는 class는 반드시 class 앞에 abstract를 붙여야 한다.

## private
- 외부에서 절대로 접근하지 못하게 만드는 키워드이다.
- 상속을 받은 자식 또한 접근하지 못하므로, super class가 가지고 있고 변경할 필요가 없는 데이터들에게 작성한다.
